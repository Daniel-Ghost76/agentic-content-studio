# Reverse Engineer Skill — How to Analyze a Reference YouTube Video

Claude executes this stage. One YouTube URL in, a full reverse-engineered analysis out. Follow these steps in order.

---

## Interaction Protocol (read this first)

This skill is intentionally interactive. Do not power through silently. Surface questions at every checkpoint below and whenever something is genuinely ambiguous. **When in doubt, ask — never guess.**

**Always ask when:**
- A visual element could belong to more than one sub-library (e.g., is this a partial overlay or a full-screen replacement? Is this a HyperFrames motion graphic or a Higgsfield AI clip?)
- A screenshot shows something unclear (blurry text, cut-off panel, ambiguous animation state)
- OCR captured text that looks malformed or incomplete
- The analysis data contradicts what the screenshots show
- A proposed new capsule is similar enough to an existing one that merging might be better
- The video duration or structure is unusual (very short, very long, non-standard format)
- Daniel's stated intent ("all sub-libraries") could mean different things given what the video actually contains

**Never guess about:**
- Exact timestamps (use the data from `_analysis.md` + screenshot filenames only)
- Whether something is AI-generated vs real footage (flag it as "unclear — needs Daniel's input")
- Design token values you cannot read clearly from a screenshot
- Whether an existing capsule is a "close match" if the difference is significant enough to warrant a separate capsule

**Checkpoints in this skill:**
- **Checkpoint A** — after Step 3: show detection summary, get go-ahead for screenshots
- **Checkpoint B** — after Step 4: show screenshot inventory, ask for any extra drill-downs
- **Checkpoint C** — after Step 5: present per-sub-library findings summary, get confirmation before mapping
- **Checkpoint D** — Step 6: present all library proposals, hard stop before writing anything
- **Checkpoint E** — Step 8: confirm each new capsule before creating it

## Step 3 — Download & Automated Detection

```bash
# Extract VIDEO_ID from URL
VIDEO_ID=$(python3 -c "
import re, sys
m = re.search(r'(?:v=|youtu\.be/)([A-Za-z0-9_-]{11})', sys.argv[1])
print(m.group(1) if m else 'unknown')
" "<URL>")

EDIT_DIR="Youtube/Output/4. Editing/reference_videos/${VIDEO_ID}"
mkdir -p "${EDIT_DIR}"

# Run video_analyst.py — downloads via yt-dlp, detects scene changes, zoom events,
# text overlay OCR (Tesseract), face detection, motion. Caches to ~/.claude/video_analyst_cache/
python3 "Youtube/Input/5. Tools/4. editing_tools/video-use/helpers/video_analyst.py" \
  "<URL>" --output-dir "${EDIT_DIR}"

# Transcribe with ElevenLabs Scribe (word-level timestamps, cached per source)
export ELEVENLABS_API_KEY=$(grep '^ELEVENLABS_API_KEY=' ~/.claude/.env | cut -d'=' -f2-)
python3 "Youtube/Input/5. Tools/4. editing_tools/video-use/helpers/transcribe.py" \
  "~/.claude/video_analyst_cache/${VIDEO_ID}.mp4" \
  --edit-dir "${EDIT_DIR}" \
  --language en

# Pack transcript into phrase-level reading view
python3 "Youtube/Input/5. Tools/4. editing_tools/video-use/helpers/pack_transcripts.py" \
  --edit-dir "${EDIT_DIR}"
```

## Step 4 — Screenshot Capture Pass (30–50 frames)

Screenshots are the primary visual classification layer. Claude reads them as images to identify glass panel styles, slide layouts, Higgsfield clip content, overlay positions, and design tokens that OCR alone cannot describe.

**Build the timestamp list:**
1. Collect all event timestamps from `{VIDEO_ID}_analysis.md` (text overlays, zoom events, scene changes)
2. Add one frame at the midpoint of each zoom hold
3. Add distribution samples every 60s of video duration (grade and style consistency)
4. Sort, remove any within 2s of another (keep earlier one)
5. If total > 50: keep all detected overlay events first, then zoom peaks, then trim distribution samples
6. If total < 30: add evenly-spaced samples to reach 30 minimum

**Extract frames:**
```bash
mkdir -p "${EDIT_DIR}/screenshots"
# For each timestamp T (in seconds):
ffmpeg -ss <T> -i "~/.claude/video_analyst_cache/${VIDEO_ID}.mp4" \
  -vframes 1 -q:v 2 \
  "${EDIT_DIR}/screenshots/frame_$(printf '%07d' $(python3 -c "print(int(<T> * 1000))"))ms.jpg" \
  -y 2>/dev/null
```

**Drill-down filmstrips for ambiguous events:**
```bash
python3 "Youtube/Input/5. Tools/4. editing_tools/video-use/helpers/timeline_view.py" \
  "~/.claude/video_analyst_cache/${VIDEO_ID}.mp4" <event_start - 0.5> <event_end + 0.5>
```

Read ALL screenshots before starting Step 5.

## Step 5 — Analysis Across 8 Dimensions

Read all of: `_analysis.json` + `_analysis.md` + `takes_packed.md` + all screenshots + any filmstrip PNGs. Do not begin writing until all images have been viewed.

For each dimension below, if the evidence is ambiguous or contradictory, **stop and ask** rather than picking one interpretation. Flag the specific screenshot filename and timestamp when asking.

---

### Dimension A — Editing Rhythm

- Cuts per minute: avg / min / max
- Typical segment duration before a cut
- Pacing: fast / medium / slow / variable
- Structure: continuous flow / clear sections / chapter-style

**Ask if:** the cut count seems surprisingly high or low (could indicate detection errors) or the structure is non-standard.

---

### Dimension B — Pre-Production Materials (Sub-library 01)

**Detection signal:** The SCREEN CONTENT ITSELF is a designed slide — not an overlay on top of footage, but the footage showing a Keynote/PowerPoint-style graphic, diagram, or designed proof screen. Face cam may be in a corner but the dominant frame is the slide.

Extract:
- Slide types: presentation / data viz / hand-drawn diagram / screenshot proof / process flow
- Layout: headline size + weight, body text size, label style, whitespace
- Colour palette (hex estimates from screenshots where readable)
- Background: dark / light / gradient / brand colour
- Diagram/illustration style: minimal clean lines / annotated / icon-based / wireframe
- Slide animation: instant / builds in / animated elements
- How often they appear and when (intro / steps / proof / outro)
- Closest 01 capsule direction: cinematic product demo / minimal thesis / plain educational / workflow diagram / proof result

**Ask if:** unclear whether a moment is a slide (pre-production material) vs a full-screen HyperFrames overlay (sub-library 03) — show the screenshot and ask Daniel which it is.

---

### Dimension C — Partial Overlays HyperFrames (Sub-library 02)

**Detection signal:** A small frosted glass panel ON TOP of footage. In corners or bottom edges. Footage visible underneath. Max ~480px wide. Does not cover the presenter's face.

Per observed type:
- Type: callout card / stat metric / step list / UI label / other
- Position: bottom-left / bottom-right / top-right / etc.
- Estimated width (px range)
- Glass panel: fill opacity, blur level, border colour
- Text specs: label size, value/number size, font weight
- Animation: entrance direction, hold, exit
- Timing trigger: spoken word / silence / topic shift
- Duration (seconds), frequency (per minute)
- Closest capsule: `02a_callout_card` / `02b_stat_metric_card` / `02c_step_list` / new

**Ask if:**
- A panel appears to cover more than half the frame — it might be a full-screen overlay (03), not partial (02)
- The glass style is noticeably different from our design system — confirm whether to treat it as a close match or new pattern
- The OCR text is cut off or hard to read — share the screenshot and ask what it says

---

### Dimension D — Full-Screen Overlays HyperFrames (Sub-library 03)

**Detection signal:** The ENTIRE FRAME is replaced by a full-screen designed composition. Footage disappears. Dark background + large centred glass card, or a node diagram, or a framed screenshot. The "liquid glass" full-screen effect.

Per observed moment:
- Type: chapter break / workflow map / proof screen / section transition / other
- Background: solid dark / frosted footage underneath / gradient
- Glass card width estimate (e.g., 720px vs 1200px centred)
- Content: heading + subtitle / diagram / screenshot in frame
- Entrance: fade in / scale 0.97→1.0 / slide up / instant
- Exit: fade / cut
- Duration (seconds), count and structural position
- Closest capsule: `03a_chapter_break` / `03b_workflow_map` / `03c_proof_screen` / new

**Ask if:** unsure whether footage is truly gone (full-screen overlay) vs footage still playing underneath (partial overlay). Show the screenshot.

---

### Dimension E — Higgsfield Visuals (Sub-library 04)

**Detection signal:** Full-screen cinematic clip — atmospheric, abstract, smooth motion, data flows, glowing elements, concept metaphors. Premium look, dark backgrounds, no presenter. Not HyperFrames — a rendered video clip.

Per observed clip:
- Concept (what's being visualised)
- Closest type: `04a_ai_tech_hud` / `04b_workflow_transformation` / `04c_abstract_concept`
- Duration (seconds)
- Colour tone: cool blue-green / warm amber / neutral dark
- **First frame description** (visual content, composition, mood) — becomes DALL-E input
- **Last frame description** (resolved state) — becomes second DALL-E input
- Transition in / out: hard cut / fade / dissolve
- Total count, where in video

**Ask if:**
- Unclear whether a cinematic moment is AI-generated or real stock footage — ask Daniel if he can tell from watching the reference video
- The first and last frame descriptions are ambiguous — show the screenshots and describe what you see, ask Daniel to confirm before writing them down

---

### Dimension F — Emphasis Word Overlays (Sub-library 05)

**Detection signal:** Large styled text ON SCREEN at high-impact speech moments. NOT regular subtitles. Large (80–140px), designed, very sparse (1–3 per video max).

Extract:
- Types: punch word / two-word contrast / quote highlight
- Exact text content
- Font size estimate, weight, colour
- Position: centre / slightly above centre / lower third
- Animation: slam + scale / fade / word-by-word reveal
- Hold duration, total count
- Which speech moment triggered each
- Closest capsule: `05a_punch_word` / `05b_two_word_contrast` / `05c_quote_highlight` / new

**Ask if:** unsure whether something is an emphasis word overlay vs a regular caption/subtitle vs part of a full-screen overlay. Show the screenshot.

---

### Dimension G — CTA Subscribe Visuals (Sub-library 06)

**Detection signal:** Subscribe prompts, next-video teasers, comment prompts, like reminders anywhere in the video.

Per CTA:
- Type: subscribe ask / next video tease / comment prompt / like reminder
- Position: bottom-centre / bottom-right / centred
- Style: pill / glass card / full overlay / badge
- Animation in/out, hold duration
- When in video: first 30s / mid / outro
- Tone: quiet/native / bold/attention-grabbing
- Closest capsule: `06a_subscribe_ask` / `06b_next_video_tease` / `06c_comment_prompt` / new

**Ask if:** the CTA style is notably different from all three existing capsule types — describe what you see and ask if it warrants a new capsule.

---

### Dimension H — Color Grade, Zoom, and Caption Style

**Color grade:**
- Warmth, contrast, saturation (from multiple clean face-cam screenshots)
- Skin tone treatment
- Closest preset: `warm_cinematic` / `neutral_punch` / `none` / custom

**Zoom behaviour:**
- Scale peaks observed
- Trigger pattern: payoff word / topic pivot / section start / random
- Typical ease-in + hold + ease-out
- Frequency vs our max 1 zoom per 60s rule

**Caption / subtitle style:**
- Word grouping, case, position, font size + weight estimate, background treatment
- Sync precision: tight (≤100ms) / loose (200ms+)

**Ask if:** the grade is hard to assess from the screenshots (heavy LUT, non-standard colour) — note it as "unclear, sample these timestamps manually."

## Step 6 — Library Mapping

For every pattern in the confirmed analysis, assign ONE classification:

**Exact match** → existing capsule fits as-is → record slug

**Close match** → existing capsule with a specific change → write the exact line change:
`"Update 02b_stat_metric_card/prompt.md: change number hero size from 72px to 96px"`

Store close-match changes in `{EDIT_DIR}/{VIDEO_ID}_proposed_library_updates.md`

**New pattern** → no existing capsule → generate full draft spec:
```
Sub-library: {01/02/03/04/05/06}
Capsule slug: {N}{letter}_{descriptive_name}

README.md: when to use | what good looks like | reference: this video at {timestamp}
prompt.md: variables list | full prompt text from observed pattern
notes.md: Status: draft — seeded from {VIDEO_ID} | Tests run: 0 | Open questions: [list]
```
