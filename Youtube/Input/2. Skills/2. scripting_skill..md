# Scripting Skill

Claude executes this stage by default.

## What This Skill Does

Runs a full interactive scripting session: idea selection → structure interview → beat-by-beat interview → Draft 1 → section-by-section review → Draft 2 PDF. Produces a Visuals Brief MD for the Codex pre-production materials agent when the video needs slides or screen overlays.

Also handles **Phase 7 — Description YAML Generation**: a separate invocation (after editing is complete) that reads the approved script and the Review transcript/subtitles, then produces `{project_id}_metadata.yaml` in `Youtube/Output/6. Review  /{project_id}/` ready for the publishing agent.

## Process

### Phase 0 — Idea Selection

List the next 3 unpublished ideas from `Youtube/Output/1. Ideation/` as:
- Title · video type · one-thing · scripted status

Present to Daniel and wait for selection. Read the full idea PDF for the chosen video.

---

### Phase 1 — Structure Interview → Format Card

Use the AskUserQuestion panel tool for every interview question — never ask in plain chat text. Group up to 4 questions per call. Use multiple choice options (2–4) where possible; "Other" is always available.

Questions to cover:
1. Video type — concept explainer / tutorial-build / news demo / story-business
2. Live demo? What exactly, what does it look like on screen?
3. Screen sharing? What are you sharing?
4. Non-demo sections: talking head only or slides/visuals?
5. Camera framing — full face / face + screen PIP / screen-only / mix?
6. Beat transitions — cut-to-cam / b-roll / stay on screen?
7. Additional constraints beyond the brief?

After all answers, write and post the Format Card (bullet list). Do not proceed until Daniel confirms it.

---

### Phase 2 — Deep Dive Interview (Beat by Beat)

For each beat in the idea brief:
1. Summarise the beat in 2–3 plain-language sentences
2. Confirm the angle or ask what to adjust
3. Extract the specific example or moment Daniel will use
4. Confirm what's on screen during this beat
5. Get pacing intent — tight / standard / room to breathe
6. Get the transition out

For [DANIEL TO FILL] beats (Personal Angle, Vulnerability):
- Read the suggested prompt from the brief
- If the answer stays abstract, push back once: ask for the specific moment — what happened, when, what was said or done

Do not start Phase 3 until every beat is mapped with real content.

---

### Phase 3 — Draft 1 + Visuals Brief

**A. Script Draft 1 — in-chat only**

Write the full script using the standard production format:
- YAML frontmatter block
- Production legend
- Each beat: TH/LIVE tag + beat title + word count target + script body
- Visual cues inside the body: `*[Screen: ...]*`
- Delivery annotations in styled span
- Thesis line in highlighted block: "Slow down. This is the line."
- Voice checklist at end
- Read-through notes at end

Run a rough word count on the script body (exclude frontmatter, checklist, notes). If more than 15% over or under the pacing target for this video type, revise before posting.

Post the draft in-chat clearly labelled `--- DRAFT 1 ---`. No script file is created at this stage.

Immediately after Draft 1 exists, create empty Output project folders for this `project_id` across all nine stages:

```text
Youtube/Output/1. Ideation/{project_id}/
Youtube/Output/2. Scripts/{project_id}/
Youtube/Output/3. Pre-production Materials/{project_id}/
Youtube/Output/4. Editing/{project_id}/
Youtube/Output/5. Visuals/{project_id}/
Youtube/Output/6. Review  /{project_id}/
Youtube/Output/7. Publishing/{project_id}/
Youtube/Output/8. Distribution/{project_id}/
Youtube/Output/9. Analytics/{project_id}/
```

**B. Visuals Brief MD — only if slides/visuals confirmed in Phase 1**

Read the format spec at: `Youtube/Input/4. Resources/3. pre-production_materials_resources/visuals_brief_format.md`

Save to: `Youtube/Input/4. Resources/3. pre-production_materials_resources/{project_id}_visuals_brief.md`

Tell Daniel the file is saved and the path before moving to Phase 4.

Skip entirely for pure demo/tutorial videos.

---

### Phase 4 — Section-by-Section Review

Strict sequential flow — Beat 1 through to the last beat, in order, no jumping:

For each beat:
1. **Paste the full beat text in chat** — so Daniel can read it at full size. Never put script text inside AskUserQuestion panel options.
2. **Open one AskUserQuestion panel** for that beat — options: Approve / Change the tone / Change the example / Change the length / Rewrite completely
3. If changes are needed: propose the specific edit in chat, then open a follow-up panel to confirm
4. Once Daniel confirms — move to the next beat. Never return to a confirmed beat unless Daniel explicitly asks.
5. Never group multiple beats into one panel. One beat at a time, every time.
6. Never skip a beat, even if it looks clean.
7. Propose specific edits — never ask "what should I change?"
8. If Daniel approves a weak beat without feedback, push back once with a targeted question.

Do not begin Phase 5 until every beat is confirmed in sequence.

---

### Phase 5 — Draft 2 HTML → PDF → Clean Up → Save

1. Apply all confirmed changes from Phase 4
2. Write the full HTML using the same production format as Phase 3
3. Convert to PDF:

```
"/Applications/Google Chrome.app/Contents/MacOS/Google Chrome" \
  --headless --disable-gpu \
  --print-to-pdf="/absolute/path/to/output.pdf" \
  --no-margins \
  "file:///absolute/path/to/input.html"
```

4. Confirm the PDF is written and readable
5. Delete the HTML immediately
6. Save to: `Youtube/Output/2. Scripts/{project_id}/{project_id}_script.pdf`
7. If screen content changed during Phase 4, update the Visuals Brief MD before saving

---

### Phase 6 — Approval

- Post a short summary: what changed from Draft 1 to Draft 2 (2–4 bullet points max)
- Ask for final approval
- On approval: session closed, script locked, ready to record
- If Visuals Brief exists, remind Daniel of its path in `Youtube/Input/4. Resources/3. pre-production_materials_resources/`

---

### Phase 7 — Description YAML Generation (Separate Invocation)

**Trigger:** `/description` command, or (future) file watcher when an SRT/MP3/transcript file lands in Review. This is NOT an automatic continuation of Phase 6 — it runs later, once editing is done.

**Inputs to read:**
- The `.srt` file in `Youtube/Output/6. Review  /{project_id}/{project_id}_review_subtitles.srt` for this video — use this for all timestamps
- The `.txt` transcript in the same folder — use this to extract tools, links, and section content
- Script PDF if available: `Youtube/Output/2. Scripts/{project_id}/{project_id}_script.pdf`

**File naming rule:**
Use `{project_id}_metadata.yaml`. Never use a different name. Never use `.yml`.

**Save to:** `Youtube/Output/6. Review  /{project_id}/` (the review agent prepares the publishing pair from there)

**What to extract from the transcript:**
- Video title — craft it based on content (max 100 chars); never use generic placeholders
- All tools, platforms, services named in the transcript — with URLs where known
- Any codes or links mentioned verbatim
- Real numbers said in the video (use them — "502 videos", not "hundreds")

**Parsing the SRT for timestamps:**
1. Read the SRT file top-to-bottom
2. Identify only the major topic shifts — not every sub-step or demo beat
3. Scale count to video length: ~10 min → max 5 timestamps; shorter → fewer; never exceed 5 for anything under 15 minutes
4. Use the timecode from the SRT entry where that section begins
5. Format: `M:SS – {Section Name}` — no leading zeros on minutes, e.g. `3:01` not `03:01`
6. First entry is always `0:00 –` with the hook/intro label
7. Section names should be plain descriptive labels, not beat names from the script
8. **After selecting all entries, verify every consecutive gap is ≥ 60 seconds.** If any two are closer than that, drop the less significant one. YouTube silently rejects the entire chapter set if any chapter is under 10 seconds; the 60-second floor keeps well clear of that and produces a clean, readable chapter list.

**Write the YAML file:**

```yaml
title: "..."

description: |
  [Hook line — first 150 chars; curiosity gap or bold claim; never "In this video I show you"]

  [2–3 sentence body: outcome-first. What the viewer walks away knowing or able to do.]

  TOOLS USED:
  - [Tool name]: [URL if known, or descriptive note]

  TIMESTAMPS:
  0:00 – [Hook / intro label]
  x:xx – [Section name]
  ...

  #Hashtag1 #Hashtag2 [...8–12 total, broad → niche order]

category_id: "28"
made_for_kids: false
```

**Section header rule:** Use `TOOLS USED:` for tutorial/demo videos. Use `LINKS MENTIONED:` for story/business videos where links matter more than tools.

**Confirm to Daniel:** file saved, full path, ready for review agent.

---

## Visual Cue Requirement

Every concept beat must have at least one `*[Screen: ...]*` cue. Nothing explained abstractly — always assume slides or screen recording is visible to the viewer at all times.

---

## What Good Output Looks Like

- Daniel can pick it up and record with zero prep questions
- Every technical term explained in the same sentence it appears
- Every beat has a visual — nothing floats in the air
- The framework lands at the end and feels like a payoff, not a summary
- The thesis line arrives quietly with a "Slow down" delivery annotation
- Read-through notes tell Daniel exactly where to accelerate, pause, and let breathe
- Every [DANIEL TO FILL] field has a real specific moment, not a concept
